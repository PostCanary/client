## Task 2: Reuse the existing render pipeline (do NOT build a new wrapper)

**Why:** CLAUDE.md (client root) already documents the render workflow: `/dev/postcard-preview` route + `scripts/screenshot.mjs` + `npm run screenshot:postcard`. Building a parallel `templates/render.html` with its own Google Fonts loader would drift from the live app's font stack (Oswald is already loaded globally in `src/styles/index.css`) and produce different output than what Drake sees in dev. Codex flagged this as a CRITICAL finding — one rendering path, not two.

**The approach:**

1. **Add an SVG-template rendering mode to the existing `/dev/postcard-preview` route.** Extend `src/pages/dev/PostcardPreview.vue` with a URL param `?svg=path/to/file.svg` that, when present, loads and inlines the SVG instead of the normal Vue postcard layouts. The SVG inherits the app's font environment automatically (no separate loader needed). Fallback: add a sibling `/dev/svg-preview` route if extending PostcardPreview.vue is too invasive — pick whichever has the smallest surface area at execution time.

2. **Screenshotting:** use the existing scripts unchanged:
   ```bash
   node scripts/screenshot.mjs "http://localhost:5175/dev/postcard-preview?svg=/templates/postcard-front-v1.svg" templates/out/postcard-front.png --wait 4000
   ```

3. **Font verification (Oswald):** Oswald is already imported in `src/styles/index.css` (verified in repo). Do NOT add a second loader. If Open Sans is required for body text on the postcard, add ONE `@import` line to `index.css` right next to the Oswald import. One font pipeline, one source of truth.

4. **Font verification test:** render a small test SVG with a single `<text font-family="'Oswald'" font-weight="700" font-size="42">TEST</text>` element. Oswald has distinctly narrower letterforms than system fallbacks — visually verify the condensed glyphs. If it looks wide/normal, the font didn't load (inspect the route's computed styles in DevTools to debug).

## Task 3: Document SVG subset rules for Figma compatibility

**File:** `templates/SVG-RULES.md`

**Rules (from research):**

### Allowed in SVG (confidence level noted — VERIFY ALL via Task 5 test):
- `<rect>`, `<circle>`, `<ellipse>`, `<line>`, `<polygon>`, `<path>` — HIGH confidence (basic shapes universally supported)
- `<text>` with `font-family`, `font-weight`, `font-size`, `fill`, `text-anchor`, `letter-spacing` — MEDIUM confidence (Figma's own export outlines text by default via `svgOutlineText: true`, suggesting text-as-text is fragile. Source: Figma ExportSettings API docs)
- `<linearGradient>` with simple 2-4 stop definitions — MEDIUM confidence (forum reports of breakage exist, but simple linear should work. Source: forum.figma.com gradient threads)
- `<image href="data:image/jpeg;base64,...">` — LOW confidence (no official docs confirming this. Forum thread mentions xlink:href path length issues. MUST test.)
- `<g>` groups with `id` attributes — HIGH confidence (Figma supports `svgIdAttribute` on export, likely respects on import)
- `opacity` attribute on any element — HIGH confidence
- `transform="translate(x,y)"` — HIGH confidence

### NOT allowed (breaks on Figma import — HIGH confidence these fail):
- `<radialGradient>` with complex stops — can distort (forum confirmed)
- Angular/conic gradients — not supported in SVG spec
- `filter` effects (`drop-shadow`, `blur`, etc.) — get rasterized to bitmap (Figma docs confirm. Also violates Draplin D-05)
- `<foreignObject>` — not imported (SVG spec limitation in most tools)
- `<use>` with external references — broken across tools
- CSS `@import` or `<style>` blocks with complex selectors — unreliable
- `font-style: italic` on unavailable fonts — substitution is unpredictable
- `<mask>` and `<clipPath>` with complex shapes — may flatten

### Photo embedding method:
1. Read `templates/photos/hvac-hero.jpg`
2. Convert to base64 (Windows-safe, Node one-liner):
   ```bash
   node -e "process.stdout.write(require('fs').readFileSync(process.argv[1]).toString('base64'))" templates/photos/hvac-hero.jpg > templates/photos/hvac-hero.b64
   ```
   (Do NOT use `base64 -w 0` — it's a Linux coreutils command that doesn't exist in PowerShell or Windows Git Bash by default. Node is already installed.)
3. Embed as: `<image id="hero_photo" x="0" y="0" width="648" height="432" href="data:image/jpeg;base64,{BASE64_STRING}" preserveAspectRatio="xMidYMid slice"/>`
4. File size will be ~5-7MB — acceptable for a template file
5. In Figma, this imports as an image layer that can be repositioned/replaced

### Font strategy:
- **In SVG:** Specify `font-family="'Oswald', 'Impact', sans-serif"` — cascading fallback
- **For Puppeteer render:** The render target is `/dev/postcard-preview`, which loads fonts via `src/styles/index.css`. Oswald is already imported there (verified). If Open Sans is used, add ONE `@import` line to `index.css` next to Oswald. Single source of truth for fonts.
- **For Figma:** Drake installs Oswald on Windows. Figma uses installed system fonts for text layers.
  - Install: https://fonts.google.com/specimen/Oswald → Download family → Install Oswald-Bold.ttf
- **If Drake doesn't install font:** Figma shows text with substituted font. Text is still editable, just looks different. Can fix by installing font later.
- **Token discipline (Wathan):** In the actual SVG code, every font-size, fill, x, y should reference CSS custom properties from `src/styles/print-scale.css` rather than hardcoding hex/pt values. The coordinate tables in Plan 02 document intent; the generated SVG uses `var(--color-headline-red)` etc. If `print-scale.css` is missing a token a plan needs, add it.

### Element naming convention:
Every variable data element gets an `id` matching the template variable:
- `id="hero_photo"` — hero background photo
- `id="headline_1"` through `id="headline_5"` — headline text lines
- `id="offer_amount"` — "$750 OFF" text
- `id="offer_detail"` — "a full A/C system replacement" text
- `id="phone_number"` — phone number
- `id="business_name"` — company name
- `id="logo_placeholder"` — logo image/shape
- `id="website_url"` — website
- `id="qr_placeholder"` — QR code area
- `id="cta_text"` — call to action text
- `id="rating_stars"` — star rating display
- `id="review_text"` — customer review quote
- `id="trust_badges"` — trust badge row group

## Task 5: Figma SVG import compatibility test

**Why:** Figma has NO official documentation of which SVG features survive import. The "allowed" and "not allowed" lists in SVG-RULES.md above are based on community reports and Figma's export docs (inverse inference). Before building real templates, we MUST verify with a real test.

**Research sources consulted:**
- Figma Plugin API docs (`createNodeFromSvg`) — confirms SVG import exists but no feature matrix (Source: developers.figma.com/docs/plugins/api/figma)
- Figma `ExportSettingsSVG` docs — `svgOutlineText` defaults to true on export, implying text-as-text is fragile (Source: developers.figma.com/docs/plugins/api/ExportSettings)
- Figma forum threads on gradient import issues (multiple reports of breakage)
- Puppeteer docs — `waitUntil: 'networkidle2'` confirmed for font loading, `waitForFonts: true` available in PDFOptions (Source: github.com/puppeteer/puppeteer/blob/main/docs/guides/pdf-generation.md)

**Test SVG file:** `templates/figma-import-test.svg`

**Must include ALL of these features to verify:**
1. `<linearGradient>` with 3 stops (navy to transparent — our overlay)
2. `<text>` with `font-family="'Oswald'"` + `font-weight="700"` + `font-size="42"` (our headline)
3. `<text>` with `font-family="'Open Sans'"` + `font-weight="400"` + `font-size="14"` (our body)
4. `<image href="data:image/jpeg;base64,...">` with a small test JPEG (~10KB, not the full hero)
5. `<rect>` with solid fill colors (#E53935 red, #43A047 green, #0D2B4B navy)
6. `<g id="test_group">` with named group
7. `opacity="0.85"` on a rect
8. `letter-spacing` and `text-anchor` attributes on text
